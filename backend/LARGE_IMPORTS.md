# 大文件导入的技术边界

DataMaster 桌面后端允许单个文件最高 **500 MiB**，一次导入的文件内容总计也最高
**500 MiB**。HTTP 请求层配置为 512 MiB，只是给 multipart 表头和边界预留协议开销；
业务代码仍会按文件内容严格执行 500 MiB 总上限。

## 已采用的内存保护

- 超过 1 MiB 的 multipart 内容由 Servlet 容器写入系统临时目录，不长期堆积在 Java 堆中。
- CSV 使用顺序读取。为了可靠区分 UTF-8 与 GB18030，会先顺序校验编码，再重新打开临时文件解析；
  代价是两次磁盘读取，但不会生成整文件 `byte[]`。
- `.xlsx` 先落到临时文件，再使用 Apache POI 事件模型顺序读取工作表 XML，不创建完整
  `XSSFWorkbook` 单元格对象图。
- 旧版二进制 `.xls` 仍依赖 POI 的内存工作簿模型，因此安全限制为 64 MiB；更大的 `.xls`
  必须先另存为 `.xlsx` 或 `.csv`。
- 默认最多保留 1,000,000 条明细。可通过 `DATAMASTER_IMPORT_MAX_ROWS` 调整，但提高它时必须
  同步给 Java 进程增加堆内存。
- 完全重复行检查只精确扫描前 250,000 行；日期/产品/客户缺失，以及当前已选数值字段的
  负数统计覆盖全部已导入明细。
- 本地界面单次最多选择 30 个文件，后端另有 50 个文件的防御性上限。服务最多保留 2 个
  分析会话，每个会话都可能持有完整规范化行列表。

## 仍然存在的边界

“允许上传 500 MiB”不等于任意 500 MiB 文件都一定能完成分析。图表、分组和导出需要保留规范化后的
明细；内存消耗更取决于行数、产品/客户唯一值数量和字符串长度，而不仅是压缩文件大小。POI 的
只读共享字符串表和样式表也仍会随 `.xlsx` 的唯一字符串数量增长，这不是完全恒定内存的解析器。

建议为 500 MiB 级别导入预留至少 1.5–2 GiB 临时磁盘空间（multipart 临时文件与 `.xlsx` 只读副本可能
短时并存）。若出现行数上限或内存压力，应优先：

1. 将旧 `.xls` 转为 `.xlsx` 或 CSV；
2. 按月份、业务单元拆分明细后分批分析；
3. 删除不参与分析的长文本、图片和附件列；
4. 确认机器内存充足后，再提高 `DATAMASTER_IMPORT_MAX_ROWS` 与 JVM `-Xmx`。

临时目录默认使用 `java.io.tmpdir`，可用 `DATAMASTER_UPLOAD_TEMP` 指向空间更充足的本地磁盘。
